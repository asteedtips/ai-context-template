01 — Context

The Problem Being Solved

True IP Solutions provides VoIP phone systems, integrated security, and paging for K-12 schools and county governments across North Carolina. The company's accounting inbox receives hundreds of emails per week — payment confirmations from Bill.com and REPAY, vendor invoices, E-Rate correspondence, PO issues from school districts, and helpdesk requests that arrive at the wrong address.

Before this project, email processing ran through standalone Claude scheduled tasks with rules hardcoded in a markdown file (Email-Processing.md). Each time a rule needed to change, someone had to edit the file manually, test it, and hope nothing broke. There was no dashboard, no audit trail, and no way for other staff to see what the system was doing.

The goal: build a full-featured Inbox Monitor inside the existing TIPS Portal — with a MudBlazor UI for rule management, AI-powered email classification, integrations into Zoho Books/CRM/Desk, and a background Azure Function that processes inboxes on a configurable schedule.

02 — Process

How the Scoping Interview Worked

The entire scoping process played out as a structured conversation between Albert and Claude across multiple sessions on March 13–14, 2026. Rather than writing a spec in isolation, the approach was iterative and decision-driven: Claude asked targeted questions about architecture, auth models, data storage, and integration patterns. Albert answered each one, and the plan grew organically from those answers.

The Interview Pattern

Each decision followed the same rhythm. Claude would present 2–4 concrete options (not vague possibilities) with trade-offs. Albert picked one, sometimes with modifications. The decision was immediately recorded in a numbered table, and downstream impacts were traced through the plan — updating the data model, adjusting the phased delivery, and noting integration items.

C
Decision #23 — Testing without production sandboxes. Three options: (1) Separate sandbox environments for each external system, (2) Dry Run mode at the pipeline level — everything runs except write operations get replaced with structured logs, (3) Mock-only testing with no real API calls.

Given that Zoho doesn't offer free sandboxes and Graph test tenants are painful to maintain, I'd lean toward option 2.
A
Dry run. But I also want a "Test Run Now" button that fires a one-shot dry run regardless of the inbox's saved mode. And the dashboard should show dry-run results with an amber badge so you can tell them apart from live runs.
C
Done. Added RunMode enum (Live/DryRun) to InboxConfig, IsDryRun flag on both Cosmos document models, SimulatedAction field on ActionExecutionResult, and six new test cases covering the dry-run pipeline. The "Test Run Now" button goes in the toolbar next to "Add Inbox." Updating the mockup now.

This pattern repeated 31 times. By the end, every architectural question had a recorded answer, and the plan document reflected all of them consistently — from database columns to UI components to test strategy.

04 — Decisions

31 Architecture Decisions, Recorded

Every non-trivial question got a numbered decision. Here are the highlights grouped by domain. The full table lives in the plan document.

Auth and Identity Decisions 1–4

2
Auth model?
Two Entra ID app roles: ClaudeInboxUser (own mailbox only) and ClaudeInboxAdmin (all inboxes, rule management, system config). Admin implies user — no need to hold both.
4
User-to-mailbox mapping?
UPN equals mailbox. The logged-in user's UPN is their mailbox address. Admins can add shared mailboxes through the UI. No whitelist — Graph permissions enforce access at runtime.

Data Storage Decisions 7, 22, 27

7
Where are rules stored?
SQL Server in a new scClaude schema. Five tables: InboxConfig, InboxRule, InboxRuleConditionGroup, InboxRuleCondition, InboxRuleAction. EF Core entities with full CRUD through the UI.
22
Activity log storage?
Cosmos DB for all activity data. SQL keeps config only. Cosmos stores per-email results, run summaries, AI classifications. Partitioned by inbox+month, TTL-based retention (default 90 days). Solves PII concern and fits the append-heavy query pattern.

Processing Pipeline Decisions 11–13, 23–26

11
How do rules evaluate?
Pipeline model: fetch unread via Graph → AI classification via Claude API → rule evaluation in priority order → action execution in sequence → write to Cosmos → generate summary. All matching rules fire; StopProcessing is an opt-in brake.
12
Rule condition types?
Full filter engine with condition groups. Groups connect by AND. Within each group, conditions connect via the group's operator (AND or OR). Supports nested logic. 16 filterable fields, 12 match operators including regex and between.
23
Testing without production sandboxes?
Dry Run mode at the pipeline level. Read-only operations execute normally. Write operations are replaced with structured logs. Results still write to Cosmos with IsDryRun = true. Dashboard shows them with an amber badge. "Test Run Now" button fires a one-shot dry run.

AI and Integrations Decisions 21, 29–31

21
AI classification?
Anthropic Claude API via HTTP. Classifies emails into categories (Needs Response, Follow-Up, Informational, Payment, etc.) and generates per-run natural language summaries. New ClaudeAiService in the service layer.
29
Real-time UI updates?
Azure SignalR Service (serverless mode). The Azure Function sends progress events as it processes. The Portal's Blazor pages receive live updates — processing progress, completion, and error alerts.
31
AI rule suggestions?
A "Suggest Rules" button sends the last 50 processed emails to the Claude API. Returns 2–3 rule templates based on detected patterns. Templates open in a pre-filled rule editor. No auto-creation — user reviews each one.
05 — Architecture

System Architecture

Solution Structure — Bounded Context Pattern
TIP.App.Claude.Service/              ← Business logic, Graph/Zoho API clients
  Inbox/InboxProcessingService.cs    ← Core pipeline engine
  Inbox/InboxConfigService.cs        ← CRUD for inbox configs
  Ai/ClaudeAiService.cs              ← Anthropic API: classification + summarization
  Ai/RuleSuggestionService.cs        ← AI rule templates from Cosmos history
  Monitoring/CosmosClientService.cs  ← Cosmos DB connection management
  Monitoring/SignalRNotificationService.cs  ← Real-time push to Portal

TIP.App.Claude.Service.Entity/       ← DTOs, ViewModels, Enums
  Enums/ (6 files)                   ← ConditionField, MatchOperator, RunMode, etc.

TIP.App.Claude.UI/                   ← Blazor components
  Pages/ (4 pages)                   ← Dashboard, InboxList, InboxDetail, ActivityLog
  Shared/ (5 components)             ← RuleEditor, ConditionGroupEditor, ActionEditor, etc.

TIP.App.Claude.Func/                 ← Azure Function (background processor)
  InboxProcessorFunction.cs          ← Timer trigger + SignalR progress events

TIP.App.Test.Claude/                 ← Unit + integration tests (70% min coverage)

Hosting Model

UI Layer
Blazor Server in the existing TIPS Portal. MudBlazor components. Entra ID auth with two app roles.
Background Processing
Separate Azure Function app. Timer-triggered per inbox cron. Same App Registration, Managed Identity.
Config Storage
SQL Server, scClaude schema. 5 tables for rules, conditions, actions. EF Core entities.
Activity Storage
Cosmos DB, partitioned by inbox+month. 90-day TTL. Run summaries + per-email results.
06 — Delivery

8-Phase Delivery Plan

Phases are organized into waves that can run in parallel where dependencies allow. The plan includes per-file task lists for every phase, gated checkpoints (SSDT publish, EF reverse-engineer, mockup verification), and a progress tracker with 60+ individual components.

Wave Phase What Key Files
1 Phase 1 Database schema + EF Core entities 5 SSDT tables, 5 entity classes, DbContext updates
2 Phase 2 Service Entity layer (DTOs, enums, Cosmos models) 6 enums, 6 view models, 5 Cosmos document classes
Phase 3 Shared Zoho extensions (Books + CRM) 2 new service classes, 6 entity classes
3 Phase 4 Core service layer (pipeline, AI, monitoring) 15 files: CRUD, pipeline, 4 action executors, AI, Cosmos, SignalR
4 Phase 5 Blazor UI (pages + shared components) 4 pages, 5 shared components, NavMenu + Program.cs updates
Phase 6 Azure Function (background processor) Function project, timer trigger, SignalR output binding
5 Phase 7 Testing (70% minimum coverage) 14 test files covering all service + executor paths
Phase 8 Seed data migration SQL scripts translating Email-Processing.md into DB rows
Wave parallelism matters. Phases 2 and 3 run in parallel (both depend on Phase 1 but not on each other). Phases 5 and 6 also run in parallel (both consume the service layer). This structure lets a developer (or an AI agent) work on independent tracks simultaneously.

08 — Reflections

What Made This Work

🧠
Deep Context, Not Cold Starts
Claude started every session by pulling the latest context files from GitHub, reading the last 2–3 session logs, and loading workflow-specific documents. It knew the codebase patterns (SNOUT's bounded context structure, the SettingsService convention, the IDbContextFactory pattern) before the first question was asked. That meant decisions could reference real code patterns, not abstract best practices.
🎯
Decision-Driven, Not Document-Driven
The plan grew from decisions, not the other way around. Each question had concrete options with trade-offs. Each answer immediately updated the data model, file list, and phase plan. By the time all 31 decisions were recorded, the plan was already complete — it just needed formatting, not content creation.
🔄
Persistent Memory Across Sessions
The session logging system (built in Session 3) allowed each subsequent session to pick up exactly where the last one left off. Session logs, the plan document, and context files all synced through GitHub. When the VM reset mid-work, nothing was lost. The AI agent operated more like a colleague with a shared notebook than a stateless tool.
⚡
Mockups as Validation, Not Decoration
The HTML mockups were built to validate decisions, not to look pretty. Every UI element maps to a specific architectural choice: the RunMode column validates Decision #23, the payment warning card validates the partial-payment out-of-scope item, the CRM tab's "Attach Note" workflow validates Decision #30. When a decision changed (like CRM V1 scope shifting from field updates to note attachment), the mockup was updated immediately to stay consistent.
